Introduction

In one of the previous posts I mentioned about the Vim buffer usage. Actually this new usage came upon with me when I was trying to set the read-only mode of sublime text. The tab phrase confused me at first glance, and then another buffer confused me more. Not until now did I realize how powerful is the buffer usage of vim is.
Most of the time when I am coding, several tabs under Linux terminal are opened. You can switch between them using alt + [tab number] under RedHat (I do not know how to do the same thing under other Linux versions, like Ubuntu). Well, the tabs can be massive and switch between them can be trivial. And voila, buffer can help you with this problem.
In vim, buffers are a portion of memory that’s been loaded with the contents of a file (not the original file). Nothing happens to the original file until the buffers are wirtten to it(:w). Actually it’s easy to understand the meaning of buffer literally. Initialization used is called initializer list and it is supported c++11 onwards. To ensure code is compiled, use c++11 or later -std option. Or in general, don’t use c++98.

Or you can manage the following settings under Eclipse CDT.

  1. Make a new C++ project;
  2. Once created, right click the project and go to “Properties”;
  3. C/C++ Build -> Settings -> Tool Settings -> GCC C++ Compiler -> Miscellaneous -> Other Flags. Put -std=c++0x (or for newer compiler version -std=c++11 at the end);
  4. C/C++ General -> Paths and Symbols -> Symbols -> GNU C++. Click “Add” and paste __GXX_EXPERIMENTAL_CXX0X__ into “Name” and leave “Value” blank;
  5. Hit “Apply”, do whatever it asks you to do, then hit “OK”;

Toggle a file in Sublime as readonly

Posted on 2018-05-29 | In Software

Under Linux, the most commonly used editor is Vim. Up until now, I’m so aware of that I only captured less than 1% of its full power usage. So, there is much more to learn about Vim.
It may occur to you, that in many cases you just want the editor ignore any changes and prevent saving anything you accidently triggered. In vim you can just use the -M parameter:

While under Windows, the most popular editor is, I cannot say 100%, Sublime. What if you want the same setting as in Vim when you don’t want modify the file. Note that, this is not about those files that have their read-only flag set at the OS level, but about every file that users don’t intend to modify. Well, “Just be readlly careful, and don’t press any buttons” is undoubtedly a good choice, but unintentional modification can happen every when and now.
In sublime, you can simply implement an easy plugin, named “Toggle the View Read-Only”.

  1. preferences -> Package Control; (Here I assume you’ve already installed it.)
  2. Type “Install Package”;
  3. Type “Toggle the View Read-Only”;
  4. Install the plugin;

Close Sublime and reopen it. You can set this with a mouse-click on the buffer/tab, the open file menu and on the edir-area of the buffer itself.(Right mouse click) “Toggle Read Only” option can be seen.

来到北京的半年

Posted on 2018-05-27 | In life

工作感悟

不知不觉间来到北京已经半年了，这也意味着我的职业生涯已经有六个月了。2017年12月21日，第一天到公司的时间；2018年5月27日，我坐在这里写这篇感想。

半年前，初出象牙塔的我满怀希望与激情，迎接人生的一个新阶段–工作。工作伊始，每天都是元气满满：Linux，C++，SVN，Redmine，Procise，GDB，FPGA。我总是给自己制造很大的困难和挑战，然而这种困难和挑战往往是无意制造的。我曾无数次总结自己的大学以及研究生生涯，从来不是漫无目的地度过，而是像无头苍蝇般乱打乱撞。今天下午跟GWB在星巴克聊到了对未来的规划，StarBucks 的香浓咖啡味又勾起了去年在巴尔的摩的时光：

去年今日StarBucks中
人面桃花相映红
人面不知何处去
桃花依旧笑春风

心中泛滥起出国的欲望，但是在最好的年纪我错过了最好的机会，碌碌无为的大学生涯已然画上句号，成也罢，败也罢，都应往事随风由它去。GWB和我都注意到了邻桌的女生，在看一本“京城烟云”。这女生看起来就很有气质，于是正在夸夸其谈的我突然意识到：如果你是一坨shit，即便出国你还是一坨shit；如果你是一颗金子，在哪儿都可以发光。于是又想到了自己不堪的外貌，脑海里无数次浮现出于老师的一句话：鼻子嘴巴都是父母给的，给什么样就是什么样。说到底，我如此在乎这些表面的东西，可能正是因为内心的极度自卑导致的结果。以往我是一个及其不在乎这些浮夸东西的人，不知从什么时候就慢慢的被同化了。于是意识到詹姆斯能禁得住身边的诱惑需要多大的自制力和强大的内心。
前几天在知乎上搜“如何提高自律”，其中一个回答让我印象深刻。自律是发自内心的自律，是因为心中有目标才给了自己无限的约束和动力，而不是仅仅靠强迫自己。这启发我确实应该停下脚步仔细审视下自己，到底想要什么。昨天晚上跟WJC和LY去了一趟三里屯，一个我梦寐以求的地方，然而却让我异常失望。打扮艳丽的各色男女，琳琅满目的各种商品，与土头灰脸的底层人士形成了鲜明的对比。在那里我感受到了无尽的自卑，眼前这可望不可及的都市生活。尤记得当时在邯郸就曾有感叹：我与这车水马龙的城市不搭。回不去的故乡，融不入的城市。后来我分析原因，大城市的包容性太强，在短短的100米内，你可以遇见开劳斯莱斯的阔绰老板，也可以遇到时尚潮流的青年男女；你可以遇到肥头大脑的黑车司机，也可以遇到黝黑憨厚的环卫工人。夜晚绚烂的霓虹灯激起了无数年轻人的意气风发，也磨平了无数年轻人的梦想棱角。这对于患有重度选择困难症的我来讲简直是一场灾难。一遍庆幸自己没有沦落到“底层人士”，一边臭骂不公的人生。
在星巴克短短的四十分钟，我的思想发生了180°急转弯。把自己当下的生活规划好、过好才是最实在的选择。

II. 使用动态库

动态库的使用要复杂些，也更容易出错，尤其对于我这样的菜鸟。动态库的优点：可执行文件很小，软件更新时只需要更新可执行文件，如果第三方库没有更新，则不需要更新其中与之相关的代码；缺点：频繁的调用第三方包会降低软件运行的速度；即便如此，在我所理解的范围内，显然动态库要比静态库的优点更明显。

如果直接执行可执行文件，会出现以下报错信息：

错误出现的原因在于：可执行文件执行时无法找到相应的动态库文件；有些人会有疑问了，编译的时候不是通过 -L -l 分别指定了路径和库文件名么，为什么执行时会报错呢？
这里的重点就在于编译时和执行时，正是由于动态库编译时并没有把库文件集成进可执行文件中，所以执行的时候才需要动态库文件，而执行时的搜索路径并不依赖于编译时的路径。（这也是可以理解的，编译往往在软件发布者层面，而执行是在用户层面，两者的库文件路径肯定不在同一个目录下，所以需要分别指定库文件路径。）
解决方法：

  1. 把可执行文件和库文件放在同一个目录下；（虽然很蠢，但是确实是一种做法啊[捂脸]）
  2. 设置 LD_LIBRARY_PATH 变量并把动态库的路径添加进此变量中；

Note:
-L/lib/path -lname 也适用于静态库，当一个目录下存在同名的静态库和动态库时（e.g. libsatchel.a, libsatchel.so），ld 会优先使用动态库。所以想要使用静态库的时候要特别注意这种隐藏的规则。

后记

之前把软件的搞崩溃就是因为我在本地设置了 LD_LIBRARY_PATH 变量但是提交到 svn 后大家并没有此环境变量，负责人当时“大惊小怪”的贴了一大堆报错信息“吓唬”我，问我用了动态库还是静态库，怎么编译生成的，我当时真的一脸懵逼，对这些东西本来就不熟悉，照着网上搜的教程比葫芦画瓢设置变量等等，软件无法运行也耽误了大家的时间。其实问题的原因很简单，就是没有在公用环境中设置这个变量，（想设置我也没有权限啊~~~）。总之，吃一堑长一智，这块的问题困扰我很久了，慢慢积累了一些后权当是写给自己看的笔记了。
